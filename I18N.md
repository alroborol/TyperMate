Internationalization (i18n)

How this site handles translations

- Translations live in `_data/locales/<lang>.yml` under the `site` key.
- Pages set the language with `lang: <code>` in the front matter (e.g., `lang: en`, `lang: es`).
- The layout reads the translations and exposes them as `t` (e.g., `{{ t.download_label }}`).
- For per-language pages, create a directory for each language (e.g., `es/`) and add translated `index.md` or other pages with `lang: es`.
 - Translations live in `_data/locales/<lang>.yml` under the `site` key (used for layout-level strings).
 - Pages should have full language-specific Markdown and set `lang: <code>` in front matter (e.g., `lang: en`, `lang: es`).
 - The layout reads the translations and exposes them as `t` (e.g., `{{ t.download_label }}`) for layout-level UI (footer, language names).
 - For content, create per-language pages under language directories (e.g., `es/index.md`) with translated content.

Adding a new language

1. Add `_data/locales/<code>.yml` with the keys used in the layout and pages.
2. Add the language code to the `langs` array in `_config.yml`.
3. Create translated page files under `/<code>/` (e.g., `fr/index.md`) and set `lang: fr` in front matter.

Notes about the language switcher and redirect

- The header includes a language switcher that attempts to preserve the current page path when switching languages (it assumes page paths mirror each other, e.g., `/es/features` corresponds to `/features`).
- On the site root, a small script redirects first-time visitors (based on browser language) to the preferred language, unless the user has chosen a language previously (stored in `localStorage` as `typermate-lang-override`).

Notes

- We currently inject translation keys inside front-matter `sections` using Liquid; be careful when editing those strings.
- For automated workflows or large-scale translations, consider using a translation service to produce the locale YAML files.
